# 2026-03 上海 AI 演讲完整笔记（改版 v2）

## 1. 演讲定位
- 暂定题目：**AI Agent 的道与术：AI 开发序章落下时的工程师实践**
- 场景：2026 年 3 月底，上海
- 时长：40 min + Q&A 5 min
- 核心目标：把 AI 从“工具升级”讲成“工程范式迁移”，并给出团队可执行的方法。

## 2. 结论先行（开场 30 秒）
- 本场是“道与术双核”，不是工具横评。
- **道**：先回答约束变了什么（时间点、资源约束、协作约束）。
- **术**：再回答怎么落地（任务切分、上下文工程、质量循环、沉淀机制）。
- 最重要变化：开发者正在从“写实现”转向“编排系统与验收结果”。

## 3. 叙事主线（建议全程保持）
1. 为什么是现在：先定位时间点，再谈方法。
2. 约束是什么（道）：`context = 内存`，人类注意力稀缺，多智能体协作成为常态。
3. 方法是什么（术）：把任务、上下文、验证、沉淀做成闭环。
4. 下一跳是什么：从人工监督走向无人职守。

## 4. 40 分钟正文（按结构改版）

### 0) 开场：为什么是现在（3 min）
关键句：我们要解决的，不再是“怎么更快写代码”，而是“怎么稳定交付正确结果”。
- AI 不是加一个新工具，而是改变了工程系统的瓶颈位置。
- 回顾阶段不是怀旧，而是为了定位当前坐标并推断下一阶段。
- 本场输出：一套道术双核的工程实践框架。

过渡：先看时代分层，确认我们今天到底站在第几层。

### 1) 时代判断：从前、当下、将来（6 min）
关键句：同一件事可以用两种视角看，机制视角决定方法，工具视角帮助沟通。

机制三阶段（主讲）：
- 第一阶段：`Tab` 自动补全时代，AI 开始进入编码动作本身。
- 第二阶段：开发者通过同步“提示-响应”循环指挥 agent。
- 第三阶段（正在到来）：agent 在更长时间尺度、以更少人类指令完成更大任务。

工具四阶段（辅助表达）：
- 补全时代 -> Chatbot 时代 -> AI Editor 时代 -> Code Agent 时代。

当下判断（2025/12 ~ 2026 初）：
- 模型能力台阶式提升后，局部场景里人的串行决策速度开始成为瓶颈。
- 工程师价值重心迁移到：定义问题、设边界、做取舍、做验收。

过渡：如果时间点判断成立，下一步就要看“约束”而不是“功能清单”。

### 1.5) 现实证据：趋势与规模（3 min）
关键句：先证明变化发生，再谈方法论才有说服力。

三图三句话：
- PR 趋势：产出变快，但 review/验收/质量控制形态重构。
- Issue/backlog 趋势：启动成本下降，不等于自动清 backlog。
- AI 高使用占比趋势：先出现高手飞轮，再看能否组织化扩散。

规模背景（脱敏，单页可讲完）：
- 实现语言：Swift + Objective-C
- 代码行数：约 235 万
- 模块数：约 650
- 文件数：约 25,000

展示边界：
- 只展示趋势，不给绝对值。
- 完全脱敏，不可回推。

过渡：证据层成立后，进入“道”。

### 2) 道：新世界的底层约束（10 min）

#### 2.1 约束一：`context = 内存`（4 min）
关键句：在 agent 开发里，上下文是最贵资源，不是免费的聊天历史。
- `context` 是有限、紧张、需要预算的内存。
- 关键问题不是“喂多少”，而是“喂什么、按什么顺序喂”。
- 真正风险：上下文漂移、跨 session 断裂、关键决策丢失。

#### 2.2 约束二：Memory + Skill 决定复利（3 min）
关键句：没有记忆系统，成功经验无法跨任务复用。
- 个人 memory：偏好、约束、踩坑、决策理由。
- 团队 memory：把个人技巧沉淀成可复用资产。
- skill 化目标：把一次成功转为模板、runbook、脚手架。
- 组织挑战：如何让少数高手带动多数同事，而不是形成 AI 格差。

#### 2.3 约束三：协作范式在变（3 min）
关键句：当前在卷“人指挥多 agent”，下一阶段是“多 agent 自行协作”。
- 人类注意力稀缺，手动编排多个 agent 是过渡态。
- 关键跨越：从人工监督到无人职守。
- 典型触发：`webhook`、`cron`、事件驱动流水线。
- 趋势判断：主力 agent 逐步从 local 迁移到云端持续运行环境。
- OpenClaw 启发：随时随地可达的 agent 形态，强化了云端化方向。

过渡：约束明确后，进入“术”，即如何把系统搭起来。

### 3) 术：企业可落地的实践框架（10 min）

#### 3.1 任务切分与交付闭环（3 min）
关键句：让 agent 接受“可验收工作单元”，不是模糊需求。
- 先定义交付物：PR / patch / demo / 回复草稿。
- 再定义验收：测试、日志、回滚、风险列表。
- 工程师角色：从“亲自实现”转向“任务编排 + 质量守门”。

#### 3.2 上下文工程与 harness 策略（3 min）
关键句：效果差异主要来自上下文与执行容器，不来自提示词技巧。
- 不绑定单一 harness，允许并鼓励多 harness 并存。
- 可控注入源优先：README、架构图、关键文件、约束清单。
- 无关历史要隔离，避免污染模型注意力。
- 模型策略要防止“省成本导致错误学习”。

#### 3.3 质量循环与工程价值重排（4 min）
关键句：AI 变强后，不是“少做工程”，而是“重排工程优先级”。

部分旧实践在过时或弱化：
- 过去被视为不经济的提早优化，在部分场景反而更重要。
- feature/UI 测试的维护成本下降，自动化回归边界扩大。
- DRY 的边际价值下降，局部重复换取可理解性有时更优。

部分既有实践更加重要：
- 可观测性：测试覆盖、数量与质量指标。
- TDD：缩短试错回路，提高可验证性。
- benchmark：把性能与成本讨论从“感觉”变成“数据”。
- 可验证循环：变更 -> 证据 -> 回归 -> 沉淀。

过渡：讲完术，不靠口号，直接看两个案例。

### 4) 双案例：从高层协作到具体故障（6 min）

#### 4.1 高层级案例：大规模代码库中的 subagent 协作（3 min）
- 场景：多语言、多模块、超大仓库，单 agent 很难稳定覆盖。
- 方案：按模块/职责拆分 subagent（检索、改动、验证），主 agent 汇总与决策。
- 价值：并行度提升、责任边界清晰、协作模板可复制。
- 连接道术：这是“注意力稀缺 + context 有限”下的工程化解法。

#### 4.2 具体案例：SwiftUI / iOS 18 NavigationStack 假死（3 min）
- 现象：2~3 层 `NavigationStack` + 状态刷新，可能触发 UI 假死。
- 闭环：现象 -> 最小复现 -> agent 介入定位/复现/验证 -> workaround。
- 产出：最小复现工程、规避策略、团队 memory 记录。
- 讲法重点：不是“AI 修 bug”，而是“排障流程标准化并可复用”。

### 5) 收束：工程师的新手艺（2 min）
三句收束：
- 把问题讲清楚（定义、约束、验收）。
- 把系统搭到 agent 能发挥（上下文、可观测、自动化）。
- 把成功路径变成团队资产（memory + skill + 模板）。

## 5. Q&A（5 min）
建议优先问题：
1. 企业权限与安全如何落地？
2. 模型与成本如何分层治理？
3. 如何从人工监督平稳过渡到无人职守？
4. 团队 memory/skill 最小可行版本怎么搭？
5. 如何避免个体飞轮与团队断层？

## 7. 演讲边界与风格
- 不做无数据支撑的远期预测。
- 不把个体经验包装成普适真理。
- 每段都遵循：`观点一句话 -> 证据/案例 -> 可执行动作`。
- 语气保持工程实践导向，避免抽象口号。
