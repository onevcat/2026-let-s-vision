# 2026 上海 AI 演讲观点池（Current）

> 来源：当前整合稿 + 三份原始笔记（去重后保留）。

## 1) 主题与总主张
1. 这场分享的主题是“AI Agent 的道与术”，核心不是工具清单，而是工程范式迁移。
2. AI 开发已经从“会聊天的助手”进入“可执行的同事/执行者”阶段。
3. 2025 年后，团队主要矛盾从“写代码慢”迁移到“上下文正确性、流程可验收性、经验可沉淀性”。
4. 目标是 Agentic Engineering，而不是 Vibe Coding。
5. 道与术是双核：道负责约束与判断，术负责落地与复用。

## 2) 时代判断与趋势
6. 回顾时代判断的目的不是怀旧，而是定位我们所处时间点：理解从前和当下，才能推断将来。
7. 三阶段视角（强调机制变化）：Tab 自动补全开启第一时代；开发者通过同步「提示-响应」循环指挥 agent 是第二时代；当前第三时代是 agent 在更长时间尺度、以更少人类指令独立完成更大任务。
8. 四阶段视角（强调工具形态）同样成立：补全时代 -> Chatbot 时代 -> AI Editor 时代 -> Code Agent 时代。
9. 2025/12 到 2026 年初模型能力出现台阶式提升，人类工程师开始成为流程瓶颈（当前讲稿观点）。
10. 工程师价值重心正在迁移：写实现成本下降，问题定义/取舍/设边界更稀缺。

## 3) 现实证据与规模背景
11. 谈方法论前要先证明“变化已发生”。
12. PR 趋势传达的核心是：产出变快，但 review/验收/质量控制形态也在重构。
13. Issue/backlog 趋势传达的核心是：启动成本下降，不等于自动降低堆积。
14. AI 使用占比趋势传达的核心是：先出现“高手”(10x 甚至 100x 工程师)，是否能扩散为组织能力。
15. 所有团队数据展示必须只讲趋势，不讲绝对值。
16. 所有团队数据展示必须完全脱敏，不可回推到个人或具体项目。
17. 规模背景（脱敏）可用来建立“问题难度感”：Swift + Objective-C、约 235 万行、约 650 模块、约 25,000 文件。

## 4) “道”层观点（底层约束）
18. 软件工程瓶颈从“代码/文档同步”迁移到“上下文窗口 + 跨 session 连续性”。
19. `context = 内存`，而且是有限、紧张、最珍贵的资源，需要规划与预算。
20. 新问题是如何让 Agent 每次拿到正确上下文，处理正确尺寸的问题，且不丢关键决策。
21. 没有 memory system，就没有 AI 协作的长期复利。
22. 个人 memory system 的价值是减少重复沟通与重复踩坑。
23. 团队 memory system 的价值是把个人经验变成组织资产。
24. AI 时代常见组织问题是“少数人飞起、多数人观望”的效率断层。团队实践最重要的是如何让少数人带动多数人。
25. 2A（Develop for Agent）是新约束：Agent 正在成为软件的新用户。
26. 人类要的是结果，不是软件操作过程。
27. 平台能力、权限审计、可观测性、流程自动化会成为企业竞争关键。

## 5) “术”层观点（工程实践）
28. 实践起点是任务切分：从“写代码任务”改为“可验收交付单元”。
29. 交付物要先定义：PR、patch、可复现 demo、回复草稿。
30. 验收标准要前置：测试、日志、回滚、风险列表。
31. 工程师角色从“亲自实现者”转向“任务编排 + 质量守门”。
32. 不宜与单一 harness 绑定；应习惯并容许多 harness 并存，必要时打造适合自身团队的 harness。
33. 上下文工程的关键是“喂对信息，不喂多信息”。比如 AGENTS.md 等保持精确精简，只提及 proj domain 知识而不要提通识。
34. 上下文要来自可控注入源：README、架构图、关键文件清单、约束清单。
35. 无关聊天历史会污染上下文，应与工程上下文隔离。关注模型注意力（开头部分和最近部分，中间部分是模糊的）
36. 过度为省钱选择低质量模型，常会导致“学到错误教训”，长期更贵。
37. 每次成功都可以沉淀为 skill （runbook / FAQ / 模板 / 脚手架），形成可复用路径，节省 token。
38. 一部分旧的开发实践和观点正在过时：提早优化等以前被认为不经济的做法反而更重要；feature 测试和 UI 测试的维护成本显著下降；DRY 的边际价值下降。
39. 一部分既有工程实践反而更重要：可观测性（测试覆盖、数量与质量）、TDD、benchmark、可验证的工程循环。

## 6) 案例观点
40. 案例不该只讲“AI 很神”，而应讲“流程如何被标准化和沉淀”。
41. 高层级案例：大规模代码库中应采用 subagent 协作，而不是单 agent 全包。
42. subagent 协作的价值是并行度、责任边界和复用模板。
43. 具体案例：iOS 18 中多层 NavigationStack + 状态刷新可能触发 UI 假死。
44. 具体案例的讲法应是：现象 -> 最小复现 -> Agent 介入点 -> workaround -> 写入团队 memory。
45. 具体案例要产出可复用资产：最小复现工程、规避策略、团队记录。

## 7) 组织协作与长期风险观点
46. 当前开发者普遍在卷“人如何同时指挥多个智能体”，因为开发者注意力是稀缺资源。（10x，100x开发者）
47. 下一阶段会走向“多个智能体自行协作工作”，人类更多做策略设定与异常兜底。（人均 10x）
48. 从人工监督到无人职守，是智能体工程化最关键的一次跨越。（跨过这条线，就回不了头了）
49. 无人职守智能体的常见触发方式包括 webhook、cron，以及事件驱动流水线。
50. 未来主力 agent 很可能不在 local，而在云端持续运行环境中。
51. OpenClaw 的启发是“随身、随时、随地”的 agent 可达性，这可作为云端化与无人职守趋势的佐证。
52. AI 使用能力高度个体化，个人强不等于组织强。
53. 个性化 AI 可能提高个人效率，但会加剧团队共识碎片化风险。（AI 格差）
54. 与智能体协作是持续校准边界的过程，且边界在快速移动。
55. AI 加速世界中，代码本身不再稀缺，真正稀缺的是问题理解与优先级判断。
56. 在极端自动化闭环里，人可能被挤出执行环节，只保留责任承担角色（当前草稿中的警示观点）。

## 8) 演讲边界与表达原则（元观点）
57. 不做无数据支撑的远期预测，展望部分宁可留白。
58. 数据表达坚持脱敏与不可回推原则。
59. 不把个体经验包装为普适真理，强调适用条件。
60. 讲法应遵循统一模板：观点一句话 -> 证据/案例 -> 可执行动作。
61. 收束目标是让听众能“明天就用起来”，而不是“听完觉得很对但无从下手”。
